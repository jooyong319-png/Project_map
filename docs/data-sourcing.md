# 데이터 소싱 & 약관 아키텍처

수익화(어필리에이트·광고·구독)의 법적 토대는 **"보여주는 데이터를 우리가 정당하게 쓸 권리가 있는가"** 다.
이 문서는 데이터 레이어를 **소유권 기준으로 4층**으로 나눠, 외부 API(구글·카카오·네이버)에 대한
의존을 "있으면 좋은 보강"으로 격하시키는 구조를 정의한다.

---

## 1. 레이어 모델 (소유권 기준)

| 레이어 | 출처 | 우리 권리 | 저장 | 없어지면? |
|--------|------|-----------|------|-----------|
| **L0 Base (식별·소유)** | 공공데이터포털 LOCALDATA 인허가 | 공공누리 1유형(상업적 이용 가능, 출처표시) → **사실상 우리 것** | 영구 (seed, `src:'localdata'`) | — (자체 보유) |
| **L1 좌표** | CSV 좌표(EPSG:5174) → proj4 로 WGS84 변환 | 공개 주소의 위경도(사실 정보) | 영구 | 주소로 재변환 가능 |
| **L2 인기/태그** | Naver 블로그 검색량 → 우리가 산출한 수치·태그 | **우리 가공물**(원문 미저장) | 영구 (`blog`,`tags`) | 신호만 사라짐, 목록은 유지 |
| **L3 평점·사진** | Google Places (New) | 라이선스 표시 콘텐츠, **캐시 최대 30일** | 휘발 (gcache, 14일 TTL) | 평점/사진만 빠짐, 목록은 유지 |
| **L3 맵링크** | Kakao 로컬 | place_url(딥링크) | 영구(링크), 콘텐츠 미저장 | 링크만 빠짐 |

**핵심 불변식: L0~L2만으로 앱이 굴러간다.** L3(구글·카카오)는 전부 빠져도 "목록·주소·인기순·동선"이 살아있어야 한다.
→ 구글/카카오 약관 변경·차단·요금 이슈에 서비스 전체가 인질로 잡히지 않는다.

## 2. 스키마 (현재 seed 확장 — 테이블 추가 없음)

현재 `seed` row 는 `{ id, name, region, cat, lat, lng, place_url, icon, tags, blog }` 이고
`id` 가 `k_<kakaoId>` 라 **카카오 ID 가 곧 PK** → 카카오에 종속돼 있다. 이걸 끊는다.

```jsonc
{
  // ── L0: 소유 식별자 (공공데이터) ──
  "id":        "p_3140000_...",   // PK = 'p_' + LOCALDATA 관리번호(opnSfTeamCode+mgtNo). 안정·영구
  "name":      "대려도",
  "addr_road": "서울 강남구 역삼로 118",   // 도로명전체주소
  "addr_jibun":"서울 강남구 역삼동 ...",   // 지번(선택)
  "biz_type":  "한식",            // 업태구분명 → cat 매핑
  "status":    "영업",            // 영업상태명 (폐업 자동 제외용)
  "src":       "localdata",       // 출처(라이선스 표기·재수집 추적)

  // ── L1: 좌표 ──
  "lat": 37.49, "lng": 127.03,

  // ── 분류 ──
  "kind": "food",                 // food | travel | stay
  "cat":  "한식", "icon": "🍲",

  // ── L2: 인기/태그 (우리 가공물) ──
  "tags": ["노포"], "blog": 548,

  // ── L3 연결키 (보강용 외래키, optional) ──
  "kakao_id": "10252806",         // 매칭되면 채움 → place_url 생성
  "gid":      null                // 구글 place_id (매칭 시). 평점/사진은 gcache 에서 TTL 캐시
}
```

**마이그레이션 원칙:** 기존 `k_*` row 는 버리지 않는다. LOCALDATA 로 새 `p_*` row 를 만들고,
매칭되면 기존 카카오/구글 값을 `kakao_id`/`gid` 로 흡수한다(아래 4단계). PK만 카카오→공공데이터로 승격.

---

## 3. 파이프라인

```
                 [공공데이터 LOCALDATA]   ← 무료, 공공누리(상업 OK)
                        │ scripts/buildPlaces.mjs
                        ▼
   L0/L1   ┌─ seed(p_*): 이름·주소·업태·좌표·kind ─┐   ← 소유·영구
           │                                       │
   매칭 ───┤  (이름+좌표 근접 50m)                  │
           ▼                                       ▼
   L3   kakao_id (place_url)                  gid (구글 place_id)
           │                                       │ 런타임
   L2   naver 블로그 → blog/tags (gcache 와 별개)   gcache(평점·사진, 14일 TTL, 21일 purge)
```

- **수집(배치)**: `buildPlaces.mjs` 가 L0/L1 을 seed 에 적재. (구글·평점 안 부름 → 비용/약관 0)
- **태깅(배치)**: 기존 `buildSeed.mjs` 의 네이버 로직 재사용해 `blog`/`tags` 부여.
- **평점/사진(런타임)**: 목록 상위 N개만 `cachedEnrich`(기존) 로 14일 캐시. 그대로 둔다.

---

## 4. 매칭 (공공데이터 ↔ 카카오/구글)

공공데이터엔 평점·맵링크가 없으므로 외부 ID 를 붙여야 보강이 된다.

1. **카카오**: `name` + 도로명주소 동(洞) 으로 Kakao 키워드검색 → 후보 중 **좌표 50m 이내 & 상호 유사도 최고** 1건 → `kakao_id`.
2. **구글**: 런타임에서 이미 좌표로 Places Text Search 하므로, 첫 결과 place_id 를 `gid` 로 저장(기존 `enrichWithGoogle` 흐름 안에서).
3. 매칭 실패해도 row 는 유효(L0~L2 로 표시). 평점·맵링크만 비는 상태.

> 매칭은 1회성 배치. 카카오 검색 쿼터(일 10만) 안에서 동네 단위로 점진 수행.

## 5. 라이선스 / 약관 준수 체크리스트

- [x] **Google**: 평점·사진 캐시 ≤ 30일 (gcache 14일 TTL + 21일 purge 크론, `api/purge.js`). place_id(`gid`)만 영구. 좌표/리뷰원문 미저장.
- [ ] **Google attribution**: 평점·사진 표시 시 "Powered by Google" / 사진 저작자 표기 (UI 작업 필요).
- [x] **Naver**: 검색 원문 미저장, 우리가 산출한 수치(blog)·태그만 저장 → 재판매 아님.
- [x] **Kakao**: 콘텐츠 미저장, place_url 딥링크만.
- [ ] **공공누리**: 출처표시 의무 → 푸터/약관에 "본 서비스는 공공데이터포털 LOCALDATA(행정안전부) 데이터를 활용" 명기.
- [ ] **위치정보법**: 사용자 위치 수집 기능 추가 시 방통위 위치기반서비스사업자 신고.

   동(洞) 단위 처리·재실행 이어서(nt 마킹). 네이버 25k/일 → 동 하나가 클 수 있음(역삼동 ~3천곳≈1만콜).

3. 카카오/구글 매칭 배치로 `kakao_id`/`gid` 채우기.
